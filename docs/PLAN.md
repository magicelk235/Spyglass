# DrivePeak тАФ Good Quick Look Previews for Google Workspace Files

A macOS app that gives real, useful Quick Look previews for Google Workspace
stub files: `.gdoc`, `.gsheet`, `.gslides`, `.gdraw`, `.gform`, `.gsite`.

Out of the box, hitting Space on one of these files in Finder shows nothing
useful тАФ they are 179-byte JSON stubs. DrivePeak fixes that.

---

## 1. The Problem (verified on this machine)

A real `.gdoc` on disk contains **only** this:

```json
{"":"WARNING! DO NOT EDIT THIS FILE! ANY CHANGES MADE WILL BE LOST!",
 "doc_id":"1Ypz5VJ4eL_G2T5CJsXjYOU_4HBuF4J0gn0r-dACtmcc",
 "resource_key":"",
 "email":"yehonatan.2350@gmail.com"}
```

Key facts established by inspecting real files:

- The stub has **no title** (title = the filename, minus extension) and **no
  URL** (URL must be built from `doc_id` + file type).
- All six types share the same JSON shape (`doc_id`, `resource_key`, `email`).
- macOS assigns these files a **dynamic UTI** (`dyn.ah62d4rv4ge80s3dtqq`) тАФ
  there is no registered type тАФ so our app must **declare** the UTIs itself,
  otherwise Quick Look has no extension to route to.
- The real document content lives on Google's servers; the disk file cannot be
  rendered without a network fetch.

## 2. Goals / Non-Goals

**Goals**
- Space-bar Quick Look on any of the six types shows a genuinely useful preview.
- Works **day one with zero setup** (offline rich card).
- **Upgrades** to a full rendered page-1 preview when the user signs into Google.
- Clean, small, well-bounded Swift code. Native frameworks only where possible.

**Non-Goals**
- No editing. Read-only preview.
- No full multi-page document rendering in v1 (page-1 / first-sheet is enough
  for "good preview"; can extend later).
- No Windows/Linux. macOS 14+ only.

## 3. Two-Tier Preview (graceful degradation)

The core design decision. The extension **always** produces something good, and
does better when it can.

- **Tier 0 тАФ Offline card (always works, no auth, no network).**
  Parse the stub JSON + filename. Render a clean SwiftUI card:
  - Big type icon (Doc/Sheet/Slides/Drawing/Form/Site) with brand color.
  - Document title (from filename).
  - Type label ("Google Docs"), owner email.
  - The canonical Google URL (built from `doc_id`), shown + click-to-open.
  This is instant and needs nothing installed. It is the guaranteed baseline.

- **Tier 1 тАФ Rendered preview (when signed in).**
  If a Google auth token exists in the Keychain, the extension calls the Drive
  API to fetch a real rendered image of the document, and shows that instead of
  the plain card (title/link still shown as a header). Fetch strategy, cheapest
  first:
  1. `files.get` тЖТ `thumbnailLink` (Google-generated page-1 image). One request,
     small, fast. **Primary path.**
  2. Fallback: `files.export` to PDF (Docs/Sheets/Slides/Drawing) and render
     page 1 with `PDFKit`. Heavier; used only if no thumbnail.
  - Forms (`.gform`) and Sites (`.gsite`) have no export/thumbnail тЖТ they stay
    on the Tier 0 card (with a note). That is correct, not a bug.

Auth is **never** done inside the extension (extensions are sandboxed and
short-lived). The companion app owns sign-in and writes the token to a shared
Keychain access group; the extension only reads it.

## 4. Architecture

Three targets in one Xcode project, three clean layers:

```
DrivePeak.app            (host app тАФ thin)
 тФЬтФА Onboarding / status UI (is the extension enabled? signed in?)
 тФЬтФА Google sign-in (ASWebAuthenticationSession тЖТ OAuth)
 тФФтФА Writes token to shared Keychain

DrivePeakPreview.appex   (Quick Look Preview Extension)
 тФФтФА QLPreviewingController тЖТ SwiftUI preview view (Tier 0 / Tier 1)

DrivePeakKit  (shared framework тАФ the real logic, unit-testable)
 тФЬтФА StubParser        parse stub JSON тЖТ Stub model
 тФЬтФА WorkspaceType     the 6 types: ext, UTI, icon, color, URL + API mime
 тФЬтФА GoogleURLBuilder  doc_id + type тЖТ canonical open URL
 тФЬтФА TokenStore        read/write OAuth token in shared Keychain
 тФФтФА DriveClient       thumbnailLink / export fetch (async)
```

Why a shared framework: the parsing/model/URL logic is pure and must be
unit-testable **without** Xcode UI, and is used by both the app and the appex.
It is the part most likely to have bugs, so it is the part that gets tests.

### Data flow (extension)
```
Finder Space
  тЖТ QLPreviewingController.preparePreviewOfFile(url)
    тЖТ StubParser.parse(url)              // Tier 0 data ready
    тЖТ if TokenStore.token exists:
         DriveClient.thumbnail(doc_id)   // Tier 1, async, timeout ~3s
    тЖТ hand SwiftUI view (card or image) back to Quick Look
```
The extension has a hard time budget. If the network is slow or absent, it must
fall back to the Tier 0 card immediately тАФ never hang the Quick Look panel.

## 5. The 6 types (single source of truth)

| Ext       | Product         | URL path            | Export? | Icon/Color        |
|-----------|-----------------|---------------------|---------|-------------------|
| `.gdoc`   | Google Docs     | `document/d/`       | тЬЕ PDF  | doc, blue         |
| `.gsheet` | Google Sheets   | `spreadsheets/d/`   | тЬЕ PDF  | tablecells, green |
| `.gslides`| Google Slides   | `presentation/d/`   | тЬЕ PDF  | slides, yellow    |
| `.gdraw`  | Google Drawings | `drawings/d/`       | тЬЕ PNG  | scribble, red     |
| `.gform`  | Google Forms    | `forms/d/`          | тЭМ      | list, purple      |
| `.gsite`  | Google Sites    | `sites/d/`          | тЭМ      | globe, teal       |

Encoded once in `WorkspaceType` enum. Everything else (UTI declarations, icons,
URL building, API calls) derives from this. No duplicated type lists.

## 6. UTI declaration (critical, easy to get wrong)

Because the OS sees only a dynamic UTI, the host app's `Info.plist` must both
**declare** each extension as an exported/imported type AND the extension must
advertise it handles them via `QLSupportedContentTypes`. We declare custom UTIs
(e.g. `com.drivepeak.gdoc`) conforming to `public.json` + `public.data`, tied to
the filename extension. This is what makes Quick Look hand the file to us.

This is the single most fragile part; it gets verified first (┬з9 step 1).

## 7. Build without the Xcode IDE

Xcode 26.6 is installed but not the active developer dir. We build headless:

```bash
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer \
  xcodebuild -project DrivePeak.xcodeproj -scheme DrivePeak build
```

- No `sudo xcode-select` required for building (env var is enough).
- Ad-hoc signing (`CODE_SIGN_IDENTITY="-"`) for local install; good enough for a
  personal tool. Real Developer ID only needed for distribution.
- Project file authored via a generator (`XcodeGen` spec or hand-written
  `.pbxproj`) so it is diffable in git and reproducible.

## 8. Testing strategy

- **Unit tests** (`DrivePeakKit`, runnable via `xcodebuild test`): StubParser
  against the real JSON shape (incl. missing fields, bad JSON), GoogleURLBuilder
  for all 6 types, WorkspaceType mapping. These are the logic that matters.
- **Manual QL smoke test**: `qlmanage -p <file>.gdoc` and Finder Space on a real
  stub, after `pluginkit -m` confirms the extension registered.
- Ship a `samples/` dir of tiny fake stubs (fake doc_ids) so previews can be
  tested without touching the user's real Drive.

## 9. Implementation order (each step independently verifiable)

1. **Prove the pipeline** тАФ minimal app + appex that declares the UTIs and shows
   a hardcoded "Hello" on Space over a `.gdoc`. If this doesn't light up,
   nothing else matters. Verify with `pluginkit`/`qlmanage`.
2. **DrivePeakKit core** тАФ `WorkspaceType`, `StubParser`, `GoogleURLBuilder` +
   unit tests. Pure logic, no UI.
3. **Tier 0 card** тАФ real SwiftUI preview from parsed stub. This is the shippable
   MVP: good offline previews for all 6 types.
4. **Host app UI** тАФ status screen (extension enabled? signed in?), open-file
   button.
5. **Google auth** тАФ `ASWebAuthenticationSession` OAuth, `TokenStore` тЖТ
   Keychain. (Needs a Google Cloud OAuth client id; app works fully without it.)
6. **Tier 1 render** тАФ `DriveClient.thumbnail`, wire into extension with
   timeout + fallback to Tier 0.
7. **Polish** тАФ icons, colors, empty/error states, README with setup steps.

Steps 1тАУ3 give a fully working, useful app with zero external setup. 5тАУ6 are the
optional upgrade and can land later without touching 1тАУ3.

## 10. Risks & mitigations

| Risk | Mitigation |
|------|-----------|
| UTI routing doesn't fire | Step 1 proves it before any real work. |
| Extension hangs on slow network | Hard timeout, always fall back to Tier 0. |
| OAuth/Drive scope friction | App is fully useful at Tier 0 without any auth. |
| Ad-hoc signing won't load appex | Document the one-time `spctl`/enable step in README. |
| `thumbnailLink` needs auth header / expires | Fetch server-side style with token; cache; fall back to export then card. |

## 11. Definition of done (v1)

- Space on any of the 6 file types in Finder shows a clean, branded card with
  title, type, owner, and a working "Open in Google" link тАФ **with nothing
  configured**.
- Signing in (optional) upgrades Docs/Sheets/Slides/Drawings to a real page-1
  image.
- `DrivePeakKit` unit tests pass via `xcodebuild test`.
- One-command build documented in README; committed to git in clean steps.
